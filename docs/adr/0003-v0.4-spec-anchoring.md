This ADR resolves **two coupled decisions** that govern the
`pilot-v0.4` cycle:

1. **D1 (spec anchoring) — Option 2.** `pilot-v0.4` stays anchored
   to breakout-note v0.2 commit `3b7b18af`, the same pin
   [ADR 0001](0001-v0.2-spec-anchoring.md) and
   [ADR 0002](0002-v0.3-spec-anchoring.md) recorded for v0.2 / v0.3.
2. **First-slice selection (D9) — item B (S3).** The first v0.4
   implementation slice is **S3 — Hydrodynamic-shell calibration
   per FND class** ([`docs/program-context.md` §3.1](../program-context.md);
   work-plan-v0-4 §1 item B). S1 (DLVO aggregation pre-screen)
   remains gated on its upstream breakout note that has not yet
   landed.

The two decisions are coupled rather than independent: Option 2
says "v0.4 = implementation tightenings under the v0.2 spec
envelope"; item B (S3) is the *leading* such tightening because
its upstream breakout note exists, its blast radius is small
(documentation + parameter defaults; no API change), and it
closes the v0.3 audit-gap pin shipped in
[`docs/delta_shell_calibration.md`](../delta_shell_calibration.md).
Recording both in one ADR mirrors how ADR 0002 bundled D1 and
D9 for v0.3.

D2 (forward-compat baseline) and the remaining work-plan-v0-4
decisions (D3–D7) are not resolved here. They get their own
deliberation surfaces inside the v0.4 cycle's continuation
session (Phase 26 continuation), as ADR 0002's D2–D7 did.

## Context

[`pilot-v0.3`](../release-notes/v0.3.md) closed at Phase 24 with
all seven in-scope items (K, A, B, C, F, H, J) shipped. Tag
`pilot-v0.3` is on `main` at commit `ad48b0b`. Test suite passes
`171 passed, 0 skipped`; `ruff check .` clean. The v0.3 release
note flags candidate v0.4 tightenings (S1, S3 calibration,
`lambda_se` axis promotion, additional housekeeping).

`pilot-v0.3` was anchored under [ADR 0002](0002-v0.3-spec-anchoring.md)
to breakout-note v0.2 commit `3b7b18af` (Option 2 in the ADR-0002
options list), with S2 — Stokes-Einstein corrections at sub-150-nm
radii — as the first slice. Both decisions stand as v0.3-scoped;
this ADR re-opens the same questions for v0.4.

[`docs/program-context.md`](../program-context.md) §3.1 frames v0.4
as another tactical sub-slice inside L1 (not the L1 release; that
is `pilot-v1.0`). The L1 slice menu (S1–S7) has S2 closed, with
S3 (hydrodynamic-shell calibration) as the natural housekeeping
candidate that has no upstream dependency, and S1 (DLVO aggregation)
as the layer-defining candidate that does have an upstream
dependency.

Two questions then have to be answered before any v0.4 phase work
starts:

- **Which spec does v0.4 anchor against?** ([`work-plan-v0-4.md` §0](../work-plan-v0-4.md#0-spec-anchoring-open--d1) — three options.)
- **Which slice ships first inside that anchor?** ([`work-plan-v0-4.md` §1](../work-plan-v0-4.md#1-scope-candidates-open) candidate menu — S1 / S3 / `lambda_se`-axis-promotion / S5 / housekeeping bundle.)

If either is left implicit, every per-phase decision in the v0.4
cycle inherits an unstated assumption — Pattern 11 (Spec pinning at
commit-hash precision, [findings-process.md §11](../findings-process.md#11-spec-pinning-at-commit-hash-precision))
exists to prevent.

## Decision 1 — Spec anchor: Option 2

`pilot-v0.4` stays anchored to breakout-note v0.2 commit `3b7b18af`.
Every v0.4 candidate must fit inside the existing breakout-note
envelope. Items that would require a physics-scope expansion are
pushed out of v0.4 to a parallel breakout track or a later cycle.

### Options considered

(Mirroring [`work-plan-v0-4.md` §0](../work-plan-v0-4.md#0-spec-anchoring-open--d1) labels.)

- **Option 1 — Re-anchor to breakout-note v0.3 if/when it lands.**
  `pilot-v0.4` opens only after the breakout-note repository
  (`Deep-Relaxation-Ordering/diamonds_in_water`) publishes a v0.3
  spec. Status of upstream: not yet drafted, per Phase 24 closeout.
- **Option 2 — Stay anchored to breakout-note v0.2 commit `3b7b18af`.**
  v0.4 reuses the same pin v0.2 / v0.2.1 / v0.3 used. Same
  reviewer-text invariant; same forward-compat triple as v0.3.
- **Option 3 — Hybrid.** In-scope items that fit the v0.2 spec stay
  on `3b7b18af`; items that require physics-scope expansion (S1
  once its breakout lands; S6 wall hydrodynamics) anchor to their
  own per-slice pins.

### Decision drivers

Same drivers as ADR 0002, restated for the v0.4 cycle:

1. **Traceability.** Can a future reader unambiguously identify
   which spec a v0.4 artefact ships against?
2. **Reviewability.** Can the v0.4 cycle's per-phase reviewers see
   the spec text that justifies a phase's scope without leaving the
   repo?
3. **Forward-compat impact.** What does each option imply for the
   `pilot-v0.3` → `pilot-v0.4` cache and deliverable diffs?
4. **Risk of numeric-channel churn.** If the spec moves underneath
   the cycle (Option 1) or is split (Option 3), how likely is a §5
   cache regen mid-cycle?
5. **Upstream-dependency timing.** S1 (the layer-defining slice
   most likely to need a fresh upstream pin) is owed a breakout
   note that does not yet exist.

| Driver | Option 1 | Option 2 | Option 3 |
|---|---|---|---|
| Traceability | Excellent once upstream lands; ambiguous until it does | Excellent and immediate | Acceptable; per-item pin records add overhead |
| Reviewability | Reviewer must context-switch to a sibling repo | Same `3b7b18af` text v0.3 reviewers read | Reviewer must check which item is anchored where |
| Forward-compat impact | Unknown | Same triple invariant as v0.3 (Pattern 14 zero-default extension) | Mixed; per-item discipline required |
| Numeric-channel churn risk | Medium-to-high if upstream lands mid-cycle | Low (Phase 12.1 / Phase 23 patterns reuse cleanly) | Medium |
| Upstream-dependency timing | Blocks v0.4 open on external publication | None | None for v0.4 items not requiring breakout-note expansion |

### Why Option 2

- **Upstream status.** As of Phase 26 opening (2026-05-06), the
  `Deep-Relaxation-Ordering/diamonds_in_water` v0.3 breakout note
  has not landed. Option 1 would gate the v0.4 cycle on an
  external calendar item this repo cannot drive, breaking the
  v0.3 working-tempo pattern (~7 working days against a frozen
  pin).
- **Reviewability.** Option 2 reuses the same `3b7b18af` text the
  v0.2 / v0.3 reviewers read; Option 1 forces context-switching
  to a sibling repo at a moving pin; Option 3 multiplies per-item
  bookkeeping.
- **Numeric-channel safety.** The Phase 12.1 / Phase 23
  regression-audit pattern (zero-default extension reproduces the
  prior cache to machine precision) reuses cleanly under Option 2.
  Options 1 and 3 carry meaningful churn risk.
- **Program-context alignment.** [`docs/program-context.md` §3](../program-context.md)
  treats v0.4 as another tactical sub-slice inside L1, not as a
  layer-defining release. Tightenings under the existing envelope
  are exactly what Option 2 enables; layer-defining slices that
  exceed the envelope (S1 aggregation, S6 wall hydrodynamics)
  remain deferred to their own upstream breakouts under the
  [§5 supplier-failure clause](../program-context.md#5-coupling-to-sibling-repositories).
- **Pattern 11 + ADR 0001 / 0002 precedent.** Both favour Option 2
  in the default case where upstream has not yet moved.

### Consequences of Option 2

**Positive.** v0.4 opens immediately; the working-tempo pattern
that delivered v0.3 carries forward. The forward-compat contract
inherits the v0.3 triple invariant directly. Reviewability is
maximal — same `3b7b18af` text, same audit-gap-pin convention.
Items that don't fit the v0.2 envelope are explicitly out of scope
(clean boundary).

**Negative.** Candidates that require physics-scope expansion (S1
DLVO aggregation, S6 wall hydrodynamics, S7 thermal control as a
first-class axis) are forced out of v0.4 to a sibling track or a
later cycle. The cycle is narrower than the work-plan-v0-4
candidate menu suggests. The `provisional=True` API contract from
[ADR 0002 §"API surface"](0002-v0.3-spec-anchoring.md#api-surface--the-provisionaltrue-contract)
remains in force throughout v0.4; it does not clear until S1 lands.

**Neutral.** Option 2 reuses ADR 0001 / ADR 0002 unchanged; v0.4
is "v0.3 with further implementation tightenings under the same
envelope."

### Re-pin policy if breakout-note v0.3 lands mid-cycle

ADR 0001 / ADR 0002 precedent: if breakout-note v0.3 lands during
the v0.4 cycle, the cycle's release phase re-pins to the new
commit and records the pin movement in a release-phase follow-up
note. Mid-cycle re-pinning is *not* allowed — it would invalidate
in-flight slices' reviewability anchor.

## Decision 2 — First slice: item B (S3 — Hydrodynamic-shell calibration)

The first v0.4 implementation slice is **S3 — Hydrodynamic-shell
calibration per FND class** as defined in
[`docs/program-context.md` §3.1](../program-context.md) and
enumerated as [item B in `work-plan-v0-4.md` §1](../work-plan-v0-4.md#b-s3--hydrodynamic-shell-calibration-per-fnd-class).

The v0.4 first-slice question was more open than v0.3's was. v0.3's
first-slice deliberation (recorded as D-PC-2 in
[`program-context.md` §7](../program-context.md#7-open-decisions-where-i-need-to-choose))
arrived at "S2 first, S1 follows" because S2 had no upstream
dependency and a smaller blast radius. For v0.4, the candidates
were different in shape:

### Candidates

(Mirroring [`work-plan-v0-4.md` §1](../work-plan-v0-4.md#1-scope-candidates-open) item labels.)

- **Candidate A: S1 — DLVO aggregation pre-screen.** The
  layer-defining slice. Has the largest blast radius among the
  v0.4 candidates and clears the `provisional=True` API contract
  from [ADR 0002 §"API surface"](0002-v0.3-spec-anchoring.md). **Gated
  on upstream breakout note** that does not yet exist; if pursued
  before upstream lands, must go via the [§5 supplier-failure
  clause](../program-context.md#5-coupling-to-sibling-repositories)
  with a local audit-gap-pin spec.
- **Candidate B: S3 — Hydrodynamic-shell calibration per FND class.**
  Housekeeping per [`program-context.md` §3.1](../program-context.md#31-layer-1-realism--pilot-v10)
  summary table; closes the v0.3 audit-gap pin shipped in
  [`docs/delta_shell_calibration.md`](../delta_shell_calibration.md).
  No upstream dependency; small blast radius.
- **Candidate C: `lambda_se` → §5 scan-axis promotion.** Conditional:
  only if S2 calibration in the FND band confirms working values
  consistently ≤ 0.3. If criterion holds, this is a Pattern 14
  zero-default-extension cache schema bump. If criterion fails,
  defer.
- **Candidate D: S5 — Concentration-weighted polydispersity kernel.**
  Housekeeping per [`program-context.md` §3.1](../program-context.md#31-layer-1-realism--pilot-v10)
  summary table. Bounded change to `lognormal_smear`'s weighting
  kernel. No upstream dependency; small blast radius.
- **Candidate E: Doc-fix bundle (item L + housekeeping).** Reconcile
  the S-slice-list discrepancy between [`deliverable-index.md`](../deliverable-index.md)
  and [`program-context.md`](../program-context.md), correct the
  v0.3 release notes' candidate list, and sweep stale state. Pure
  housekeeping; ~1 session.

### Decision drivers for first-slice selection

1. **Upstream availability.** Does the candidate's breakout-note
   spec exist? S1 fails this test; S3, S5, `lambda_se`-axis, and
   doc-fix all pass.
2. **Blast radius.** Smaller is better for first-slice. Doc-fix
   < S3 / S5 < `lambda_se`-axis < S1.
3. **Layer-defining vs housekeeping.** Layer-defining slices reduce
   the v1.0 punch list; housekeeping closes audit-gap pins. Neither
   is dominant. v0.3 closed S2 (layer-defining) plus several
   housekeeping items; v0.4 could mirror that pattern with S3 or
   S5 plus housekeeping, or invert it with S1 if upstream cooperates.
4. **Cycle calendar.** v0.3 took ~7 working days for one
   layer-defining slice plus six tactical items. v0.4 has the same
   calendar envelope by default; bundling S1 into v0.4 likely
   exceeds it.
5. **Sequencing convenience vs trustworthiness.** S1 establishes the
   trustworthiness flag whose absence motivates the v0.3
   `provisional=True` contract. Until S1 lands, S2's outputs remain
   gated behind `accept_provisional=True`. The longer S1 takes, the
   longer the gate stays.

### Why item B (S3) first

- **Upstream availability.** S3's spec content (literature DLS
  measurements + manufacturer data sheets per FND class) does not
  require an unwritten upstream breakout note. Candidate A (S1
  aggregation) does; under Option 2's "no calendar gating" stance,
  S1 is deferred to v0.5 or later.
- **Smallest blast radius among ready candidates.** S3 is
  documentation + parameter defaults; no API change, no §5 cache
  schema bump, no new module. Candidate C (`lambda_se` → §5 axis)
  is conditional on a calibration result; candidates D (S5
  polydispersity kernel) and E (doc-fix) are similarly small but
  do not close a v0.3 audit-gap pin.
- **Closes a v0.3 audit-gap pin.** v0.3 shipped
  [`docs/delta_shell_calibration.md`](../delta_shell_calibration.md)
  as a *provisional* table marked as audit-gap pin. S3 promotes
  the provisional table to a calibrated default range per FND
  class (bare, carboxylated, hydroxylated, PEG-functionalised),
  retiring the audit-gap pin. This is the "first cycle work that
  closes prior-cycle debt" pattern v0.2 → v0.3 already followed
  (Phase 19 closed v0.2's `T_OBS_S` / `DEPTHS_M` pins).
- **Forward-compat is straightforward.** No API change means S3 is
  trivially v0.3-compatible. The default value of `delta_shell_m`
  acquires a citation-anchored class identity but the
  zero-default reproduction of v0.2 / v0.3 stays intact.
- **Mirrors v0.3's pattern.** v0.3 closed S2 (one program-context
  slice) plus 6 tactical items in 9–12 sessions. v0.4 with item B
  (S3) plus a housekeeping bundle is the same shape.

The order is a **sequencing convenience, not an independence
claim.** S3 outputs are not absolutely trustworthy until S1 is in
place — S3 calibrates the radius shell given trusted aggregation
state; S1 establishes when the aggregation state can be trusted.
The `provisional=True` API contract from
[ADR 0002 §"API surface"](0002-v0.3-spec-anchoring.md#api-surface--the-provisionaltrue-contract)
remains in force throughout v0.4 and does not clear until S1 lands.

### What item B (S3) ships

Per [`program-context.md` §3.1 S3](../program-context.md):

- A calibrated default range for `delta_shell_m` per FND class
  (bare, carboxylated, hydroxylated, PEG-functionalised), drawn
  from open-literature DLS measurements with citation pins.
- Replacement of [`docs/delta_shell_calibration.md`](../delta_shell_calibration.md)'s
  provisional table with a citation-anchored table marked
  audit-gap-pin-resolved (parallel to v0.3's resolution of
  `T_OBS_S` / `DEPTHS_M`).
- Forward-compatible default: the user-supplied knob
  (`delta_shell_m=0`) remains; only the *default value used when
  no knob is supplied* acquires a class identity. This preserves
  v0.2 / v0.3 byte-identical reproduction at zero-default.

The S3 implementation phase (proposed Phase 27) is responsible
for the slice itself. This ADR records the *selection*, not the
implementation plan; the work plan does the rest.

### Contingency

Manufacturer data sheets for functionalised FNDs may be
proprietary or NDA-bound. Per [`program-context.md` §3.1 S3](../program-context.md),
the fallback is open-literature DLS only with affected batches
flagged as "unverified" in the simulator metadata. The repo never
blocks on proprietary data; if literature is contradictory, the
most-cited open value is picked and the conflict is flagged in
the calibration table (carrying forward the v0.3 Phase 21
treatment).

## Coupling between Decision 1 and Decision 2

The two decisions are not independent. The combinations that were
available at promotion time:

- **D1 = Option 2 + first slice = S3, S5, `lambda_se`-axis, or doc-fix.**
  Internally consistent: all of these fit the v0.2 envelope and
  reproduce v0.3 at compatibility-mode defaults. **Chosen, with
  S3 (item B) as the first slice.**
- **D1 = Option 1 + first slice = S1.** Internally consistent
  but blocked: the upstream breakout note had not landed by
  Phase 26 opening (2026-05-06), so this combination was not
  available.
- **D1 = Option 1 + first slice ≠ S1.** Inconsistent: re-anchoring
  to a v0.3 breakout note for a v0.2-envelope tightening would be
  paying the cost of Option 1 (calendar gate, context-switch) for
  no benefit. Rejected.
- **D1 = Option 3 + first slice = S1 (under per-slice pin).**
  Internally consistent but expensive: the per-slice bookkeeping
  cost ADR 0002 rejected for v0.3 returns. Rejected for v0.4 by
  the same reasoning.

ADR 0002 set the precedent of bundling D1 + D9 in a single
promotion ceremony; this ADR follows it.

## Re-pin policy if breakout-note v0.3 lands mid-cycle (post-promotion)

Same precedent as ADR 0001 / ADR 0002:

- If it lands during Phase 27+ (post-promotion), the cycle's
  release phase re-pins to the new commit and records the pin
  movement in a release-phase follow-up note. Earlier phases are
  not re-anchored mid-cycle.
- If it does not land at all, Option 2 stands and S1 is deferred
  to v0.5 (or to a parallel breakout track).

## API surface

This ADR does **not** introduce a new API contract. The
[ADR 0002 §"API surface — the `provisional=True` contract"](0002-v0.3-spec-anchoring.md#api-surface--the-provisionaltrue-contract)
remains in force throughout v0.4. If S1 lands during v0.4, that
slice's lab note is responsible for clearing the
`provisional=True` flag (or, equivalently, replacing it with a
trustworthiness channel that the design-tool entry points consult
instead). The v0.3 → v1.0 transition ADR will pin the final shape;
v0.4 either lands S1 and triggers that transition, or defers and
keeps the v0.3 contract intact.

  re-evaluates.
- **Q3.** Forward-compat baseline (work-plan-v0-4 §3 D2): anchor

  any in-scope item moves cache numerics by design)? **Decided per
  item in the work plan §1 reshape (Phase 26 continuation), not

The boundary is deliberate: this ADR answers "what spec does v0.4

only. Everything else is the work plan's job.
