This ADR scopes itself to **two coupled decisions** that govern the
`pilot-v0.4` cycle:

1. **D1 (spec anchoring) — [OPEN].** Re-anchor to a future
   breakout-note v0.3, stay on the v0.2 commit `3b7b18af`, or
   adopt a hybrid per-item anchoring scheme.
2. **First-slice selection (D9) — [OPEN].** Which S-slice ships
   first inside the chosen anchor.

The two decisions are coupled the same way they were for v0.3:
the answer to "what spec does v0.4 anchor against?" determines
which slices are available as the first implementation slice. ADR
0002's pattern (one ADR for the coupled pair) is reused.

D2 (forward-compat baseline) and the remaining work-plan-v0-4
decisions (D3–D7) are not resolved here. They get their own
deliberation surfaces inside the v0.4 cycle as they ripen.

This is a **stub**: the options are surveyed, drivers identified,
but no decision is committed. Stewardship preference is recorded
under "Working notes" below, framed as a non-binding leaning.

## Context

[`pilot-v0.3`](../release-notes/v0.3.md) closed at Phase 24 with
all seven in-scope items (K, A, B, C, F, H, J) shipped. Tag
`pilot-v0.3` is on `main` at commit `ad48b0b`. Test suite passes
`171 passed, 0 skipped`; `ruff check .` clean. The v0.3 release
note flags candidate v0.4 tightenings (S1, S3 calibration,
`lambda_se` axis promotion, additional housekeeping).

`pilot-v0.3` was anchored under [ADR 0002](0002-v0.3-spec-anchoring.md)
to breakout-note v0.2 commit `3b7b18af` (Option 2 in the ADR-0002
options list), with S2 — Stokes-Einstein corrections at sub-150-nm
radii — as the first slice. Both decisions stand as v0.3-scoped;
this ADR re-opens the same questions for v0.4.

[`docs/program-context.md`](../program-context.md) §3.1 frames v0.4
as another tactical sub-slice inside L1 (not the L1 release; that
is `pilot-v1.0`). The L1 slice menu (S1–S7) has S2 closed, with
S3 (hydrodynamic-shell calibration) as the natural housekeeping
candidate that has no upstream dependency, and S1 (DLVO aggregation)
as the layer-defining candidate that does have an upstream
dependency.

Two questions then have to be answered before any v0.4 phase work
starts:

- **Which spec does v0.4 anchor against?** ([`work-plan-v0-4.md` §0](../work-plan-v0-4.md#0-spec-anchoring-open--d1) — three options.)
- **Which slice ships first inside that anchor?** ([`work-plan-v0-4.md` §1](../work-plan-v0-4.md#1-scope-candidates-open) candidate menu — S1 / S3 / `lambda_se`-axis-promotion / S5 / housekeeping bundle.)

If either is left implicit, every per-phase decision in the v0.4
cycle inherits an unstated assumption — Pattern 11 (Spec pinning at
commit-hash precision, [findings-process.md §11](../findings-process.md#11-spec-pinning-at-commit-hash-precision))
exists to prevent.

## Decision 1 — Spec anchor: [OPEN]

`pilot-v0.4`'s spec anchor is one of three options. ADR 0001 / ADR
0002 set the precedent for what each option costs.

### Options considered

(Mirroring [`work-plan-v0-4.md` §0](../work-plan-v0-4.md#0-spec-anchoring-open--d1) labels.)

- **Option 1 — Re-anchor to breakout-note v0.3 if/when it lands.**
  `pilot-v0.4` opens only after the breakout-note repository
  (`Deep-Relaxation-Ordering/diamonds_in_water`) publishes a v0.3
  spec. Status of upstream: not yet drafted, per Phase 24 closeout.
- **Option 2 — Stay anchored to breakout-note v0.2 commit `3b7b18af`.**
  v0.4 reuses the same pin v0.2 / v0.2.1 / v0.3 used. Same
  reviewer-text invariant; same forward-compat triple as v0.3.
- **Option 3 — Hybrid.** In-scope items that fit the v0.2 spec stay
  on `3b7b18af`; items that require physics-scope expansion (S1
  once its breakout lands; S6 wall hydrodynamics) anchor to their
  own per-slice pins.

### Decision drivers

Same drivers as ADR 0002, restated for the v0.4 cycle:

1. **Traceability.** Can a future reader unambiguously identify
   which spec a v0.4 artefact ships against?
2. **Reviewability.** Can the v0.4 cycle's per-phase reviewers see
   the spec text that justifies a phase's scope without leaving the
   repo?
3. **Forward-compat impact.** What does each option imply for the
   `pilot-v0.3` → `pilot-v0.4` cache and deliverable diffs?
4. **Risk of numeric-channel churn.** If the spec moves underneath
   the cycle (Option 1) or is split (Option 3), how likely is a §5
   cache regen mid-cycle?
5. **Upstream-dependency timing.** S1 (the layer-defining slice
   most likely to need a fresh upstream pin) is owed a breakout
   note that does not yet exist.

| Driver | Option 1 | Option 2 | Option 3 |
|---|---|---|---|
| Traceability | Excellent once upstream lands; ambiguous until it does | Excellent and immediate | Acceptable; per-item pin records add overhead |
| Reviewability | Reviewer must context-switch to a sibling repo | Same `3b7b18af` text v0.3 reviewers read | Reviewer must check which item is anchored where |
| Forward-compat impact | Unknown | Same triple invariant as v0.3 (Pattern 14 zero-default extension) | Mixed; per-item discipline required |
| Numeric-channel churn risk | Medium-to-high if upstream lands mid-cycle | Low (Phase 12.1 / Phase 23 patterns reuse cleanly) | Medium |
| Upstream-dependency timing | Blocks v0.4 open on external publication | None | None for v0.4 items not requiring breakout-note expansion |

### Working notes during deliberation (non-binding)

Recorded for context, **not committed:**

- **Stewardship leaning (U. Warring, 2026-05-05):** Option 2 by
  default, unless the v0.4 cycle adopts S1 as a first slice and
  the upstream breakout note lands before Phase 26 opening. If
  upstream lands, Option 1 becomes the cleaner choice for that
  cycle; Option 3 carries unnecessary bookkeeping for a
  single-slice case.
- **Pattern 11** ("Spec pinning at commit-hash precision") plus
  **ADR 0001 / ADR 0002 precedent** both favour Option 2 in the
  default case where upstream has not yet moved.
- **Calendar consideration.** v0.3 took ~7 working days against a
  frozen pin. Gating v0.4 on an external publication (Option 1)
  re-introduces calendar risk that the v0.3 cycle deliberately
  avoided.
- **D-PC-1 status.** [`program-context.md` §2 P-1](../program-context.md#2-programme-prerequisites-and-risks)
  flags the experimental-campaign repository as the load-bearing
  unresolved dependency. v0.4 is a tactical L1 slice and does not
  require D-PC-1 resolution to proceed; this is the same reasoning
  ADR 0002 used for v0.3.

These notes are deliberation-surface text, not a decision. The
decision is made when this ADR is promoted from `Proposed (stub)`
to `Accepted` in Phase 26 opening.

## Decision 2 — First slice: [OPEN]

The v0.4 first-slice question is more open than v0.3's was. v0.3's
first-slice deliberation (recorded as D-PC-2 in
[`program-context.md` §7](../program-context.md#7-open-decisions-where-i-need-to-choose))
arrived at "S2 first, S1 follows" because S2 had no upstream
dependency and a smaller blast radius. For v0.4, both candidates
are different in shape:

### Candidates

(Mirroring [`work-plan-v0-4.md` §1](../work-plan-v0-4.md#1-scope-candidates-open) item labels.)

- **Candidate A: S1 — DLVO aggregation pre-screen.** The
  layer-defining slice. Has the largest blast radius among the
  v0.4 candidates and clears the `provisional=True` API contract
  from [ADR 0002 §"API surface"](0002-v0.3-spec-anchoring.md). **Gated
  on upstream breakout note** that does not yet exist; if pursued
  before upstream lands, must go via the [§5 supplier-failure
  clause](../program-context.md#5-coupling-to-sibling-repositories)
  with a local audit-gap-pin spec.
- **Candidate B: S3 — Hydrodynamic-shell calibration per FND class.**
  Housekeeping per [`program-context.md` §3.1](../program-context.md#31-layer-1-realism--pilot-v10)
  summary table; closes the v0.3 audit-gap pin shipped in
  [`docs/delta_shell_calibration.md`](../delta_shell_calibration.md).
  No upstream dependency; small blast radius.
- **Candidate C: `lambda_se` → §5 scan-axis promotion.** Conditional:
  only if S2 calibration in the FND band confirms working values
  consistently ≤ 0.3. If criterion holds, this is a Pattern 14
  zero-default-extension cache schema bump. If criterion fails,
  defer.
- **Candidate D: S5 — Concentration-weighted polydispersity kernel.**
  Housekeeping per [`program-context.md` §3.1](../program-context.md#31-layer-1-realism--pilot-v10)
  summary table. Bounded change to `lognormal_smear`'s weighting
  kernel. No upstream dependency; small blast radius.
- **Candidate E: Doc-fix bundle (item L + housekeeping).** Reconcile
  the S-slice-list discrepancy between [`deliverable-index.md`](../deliverable-index.md)
  and [`program-context.md`](../program-context.md), correct the
  v0.3 release notes' candidate list, and sweep stale state. Pure
  housekeeping; ~1 session.

### Decision drivers for first-slice selection

1. **Upstream availability.** Does the candidate's breakout-note
   spec exist? S1 fails this test; S3, S5, `lambda_se`-axis, and
   doc-fix all pass.
2. **Blast radius.** Smaller is better for first-slice. Doc-fix
   < S3 / S5 < `lambda_se`-axis < S1.
3. **Layer-defining vs housekeeping.** Layer-defining slices reduce
   the v1.0 punch list; housekeeping closes audit-gap pins. Neither
   is dominant. v0.3 closed S2 (layer-defining) plus several
   housekeeping items; v0.4 could mirror that pattern with S3 or
   S5 plus housekeeping, or invert it with S1 if upstream cooperates.
4. **Cycle calendar.** v0.3 took ~7 working days for one
   layer-defining slice plus six tactical items. v0.4 has the same
   calendar envelope by default; bundling S1 into v0.4 likely
   exceeds it.
5. **Sequencing convenience vs trustworthiness.** S1 establishes the
   trustworthiness flag whose absence motivates the v0.3
   `provisional=True` contract. Until S1 lands, S2's outputs remain
   gated behind `accept_provisional=True`. The longer S1 takes, the
   longer the gate stays.

### Working notes during deliberation (non-binding)

Recorded for context, **not committed:**

- **Stewardship leaning (U. Warring, 2026-05-05):** **S3 (item B)
  as first slice, bundled with item L (doc-fix) and 2–3 small
  housekeeping items**, mirroring v0.3's pattern of one
  scope-tightening slice plus tactical follow-ups. S1 is best
  served by waiting on its upstream breakout note rather than
  forcing it into v0.4 under the supplier-failure clause; the
  cycle calendar is short enough that a one-cycle wait is
  preferable to a parallel-spec-authoring overhead.
- **Alternative:** if upstream breakout-note v0.3 lands before
  Phase 26 opening, **S1 becomes the more natural first slice**
  and v0.4 becomes a layer-defining release that clears the
  `provisional=True` contract. This is the "wait-and-see at Phase
  26 opening" path.

These notes are deliberation-surface text, not a decision.

## Coupling between Decision 1 and Decision 2

The two decisions are not independent:

- **D1 = Option 2 + first slice = S3, S5, `lambda_se`-axis, or doc-fix.**
  Internally consistent: all of these fit the v0.2 envelope and
  reproduce v0.3 at compatibility-mode defaults.
- **D1 = Option 1 + first slice = S1.** Internally consistent: the
  upstream breakout note's existence is precisely what unblocks
  S1, and re-anchoring to it is the principled response.
- **D1 = Option 1 + first slice ≠ S1.** Inconsistent: re-anchoring
  to a v0.3 breakout note for a v0.2-envelope tightening would be
  paying the cost of Option 1 (calendar gate, context-switch) for
  no benefit.
- **D1 = Option 3 + first slice = S1 (under per-slice pin).** Internally
  consistent but expensive: the per-slice bookkeeping cost ADR 0002
  rejected for v0.3 returns. Justified only if v0.4 also adopts a
  v0.2-envelope item under the `3b7b18af` pin in the same cycle.

**Implication for promotion:** the Phase 26 opening session needs
both decisions on the table together. Resolving D1 without D9 (or
vice versa) leaves the coupling ambiguous and would force a
re-deliberation. ADR 0002 set the precedent of bundling them; this
ADR follows it.

## Re-pin policy if breakout-note v0.3 lands mid-cycle

Same precedent as ADR 0001 / ADR 0002:

- If breakout-note v0.3 lands before Phase 26 opening, the cycle
  re-evaluates D1 / D9 in that opening session. No mid-cycle
  re-pinning.
- If it lands during Phase 27+ (post-acceptance), the cycle's
  release phase re-pins to the new commit and records the pin
  movement in a release-phase follow-up note. Earlier phases are
  not re-anchored mid-cycle.
- If it does not land at all, Option 2 stands and S1 is deferred
  to v0.5 (or to a parallel breakout track).

## API surface

This ADR does **not** introduce a new API contract. The
[ADR 0002 §"API surface — the `provisional=True` contract"](0002-v0.3-spec-anchoring.md#api-surface--the-provisionaltrue-contract)
remains in force throughout v0.4. If S1 lands during v0.4, that
slice's lab note is responsible for clearing the
`provisional=True` flag (or, equivalently, replacing it with a
trustworthiness channel that the design-tool entry points consult
instead). The v0.3 → v1.0 transition ADR will pin the final shape;
v0.4 either lands S1 and triggers that transition, or defers and
keeps the v0.3 contract intact.

## Open questions (acknowledged, not blocking)

These remain open inside any of the three options and are routed
to other deliberation surfaces:

- **Q1.** If D1 = Option 1 (re-anchor to breakout-note v0.3) is
  taken, what is the minimum upstream content required? S1's
  τ_agg(ζ, I, pH) parametrisation is the obvious answer; is a
  partial breakout note sufficient, or must the full v0.3 land?
  **Routed to Phase 26 opening.**
- **Q2.** If D1 = Option 3 (hybrid) is taken, where does the
  per-slice pin record live? ADR 0002 implicitly assumed one pin
  per cycle; Option 3 needs an explicit register. **Routed to
  Phase 26 opening if Option 3 is chosen, else moot.**
- **Q3.** Forward-compat baseline (work-plan-v0-4 §3 D2): anchor
  to `pilot-v0.3` (default expectation) or to a v0.4 baseline (if
  any in-scope item moves cache numerics by design)? **Decided per
  item, not here.**

## Related but not decided here

- **D2 — Forward-compat baseline.** Decided per item once §1 is
  reshaped.
- **D3 — S1 in-scope question.** Coupled to D1 via the upstream
  dependency, but the in-scope/out-of-scope question is decided in
  the work plan §1 reshape, not this ADR.
- **D4 / D5 / D6 / D7 — scope-shaping decisions for S3, `lambda_se`-axis,
  S5, doc-fix.** Decided in the work plan §1 reshape. Strong
  default expectations recorded in the work plan §5 table.
- **D8 — DOI mint at v0.4.** Carries forward as resolved (hold
  to v1.0 per [Phase 16.1](../../lab_notes/2026-04-30-phase16-1-defer-doi-to-v1-0.md)).

The boundary is deliberate: this ADR answers "what spec does v0.4
anchor against, and what slice ships first inside that anchor?"
only. Everything else is the work plan's job.
