## Context

[`pilot-v0.2`](../release-notes/v0.2.md) is closed. Its scope was
anchored under [ADR 0001](0001-v0.2-spec-anchoring.md) to
breakout-note v0.2 commit `3b7b18af`, with three forward-compatible
extensions landing as zero-default audit-gap pins. The
[`pilot-v0.2.1`](../release-notes/v0.2.md#fair-metadata-patch-021)
FAIR-metadata patch added citation infrastructure but did not
re-anchor the spec.

[`pilot-v0.3`](../work-plan-v0-3.md) is being scoped. Its candidate
items (A–J in [§1 of the v0.3 scaffold](../work-plan-v0-3.md#1-scope-candidates-open))
include both *implementation tightenings* (audit-gap-pin resolution,
continuous thresholds, mesh-convergence audit) and *physics-envelope
expansions* (T-dependent thermal diffusivity κ(T), wall-hydrodynamic
correction, possibly aggregation pre-screen).

Before the v0.3 scope can be fixed in §1 of the work plan, the
spec-anchoring question has to be answered: **does v0.3 ship against a
new breakout-note v0.3 spec, against the same `3b7b18af` pin v0.2 used,
or against a hybrid?** The same Guardian gate that opened ADR 0001 now
applies to the v0.3 cycle.

If this question is left implicit, every per-phase decision in the
v0.3 cycle inherits an unstated spec assumption — the pattern Pattern 11
("Spec pinning at commit-hash precision",
[findings-process.md §11](../findings-process.md#11-spec-pinning-at-commit-hash-precision))
exists to prevent.

## Options considered

The options mirror [`work-plan-v0-3.md` §0](../work-plan-v0-3.md#0-spec-anchoring-open)
labels.

### Option 1 — Re-anchor to a future breakout-note v0.3

`pilot-v0.3` opens only after the breakout-note repository
(`Deep-Relaxation-Ordering/diamonds_in_water`) publishes a v0.3 spec
that absorbs the relevant additions (e.g. promoting `σ_geom` to a §5
axis per ADR 0001 recommendation 3, formalising the convection
side-channel per recommendation 1). Implementation tracks the new
pin.

### Option 2 — Stay anchored to breakout-note v0.2 commit `3b7b18af`

`pilot-v0.3` reuses the v0.2 spec pin verbatim. Every v0.3 candidate
must fit inside the existing breakout-note envelope. Items that would
require a physics-scope expansion are pushed out of v0.3 to a parallel
breakout track or a later cycle.

### Option 3 — Hybrid

In-scope candidates that fit the v0.2 spec stay anchored to
`3b7b18af`. Candidates that require physics-scope expansion are
split into parallel breakouts per the v0.2 plan §1 convention,
with explicit cross-references in the work plan and per-phase
ADRs.

## Decision drivers

Four drivers should be weighed against each option:

1. **Traceability.** Can a future reader unambiguously identify which
   spec a v0.3 artefact ships against? ADR 0001 set the precedent
   that this is non-negotiable.
2. **Reviewability.** Can the v0.3 cycle's per-phase reviewers see the
   spec text that justifies a phase's scope without leaving the repo?
   Repo-internal pins help; multi-repo / time-shifted pins hurt.
3. **Forward-compat impact.** What does each option imply for the
   `pilot-v0.2.1` → `pilot-v0.3` cache and deliverable diffs?
   The v0.2 cycle's machine-precision contract (Phase 12.1) was
   load-bearing; D2 inherits this question directly.
4. **Risk of numeric-channel churn.** If the spec moves underneath
   the cycle (Option 1) or is split (Option 3), how likely is a §5
   cache regen mid-cycle? v0.2 avoided this only because ADR 0001
   pinned the spec hard.

### Driver-by-option evaluation

| Driver | Option 1 (v0.3 breakout-note) | Option 2 (v0.2 commit) | Option 3 (hybrid) |
|---|---|---|---|
| Traceability | Excellent once the upstream commit lands; ambiguous until it does | Excellent and immediate (reuses ADR 0001's pin) | Acceptable; per-item pin records add bookkeeping overhead |
| Reviewability | Reviewer must context-switch to a sibling repo at a moving pin | Reviewer reads the same `3b7b18af` text v0.2 reviewers read | Reviewer must check which item is anchored where |
| Forward-compat impact | Unknown — depends on what v0.3 spec says | Same triple invariant as v0.2 (regime labels exact, ratio / bmf machine-precision) | Mixed — known invariants for v0.2-anchored items, unknown for the rest |
| Numeric-channel churn risk | Medium-to-high if upstream lands mid-cycle | Low (Phase 12.1 pattern reuses cleanly) | Medium (sub-cycle split surfaces the same risk per parallel track) |

## Consequences per option

### Option 1 — Re-anchor to v0.3 breakout-note

**Positive.** Spec authority is preserved at v0.3. Any §5-axis
promotion (e.g. `σ_geom`) becomes first-class. Future audits
trace the cycle to a single, fully-formal spec.

**Negative.** Calendar risk is unbounded: the breakout-note v0.3 has
no announced timeline, and `Deep-Relaxation-Ordering` is a sibling
project. v0.2 took ~7 working days across ~10 sessions because it ran
sequentially against a frozen spec; gating v0.3 on an external
publication breaks that working tempo.

**Neutral.** If breakout-note v0.3 lands during the cycle, this
option converges with Option 2 + a Phase 24 re-pin (the same
mechanism ADR 0001 already documents).

### Option 2 — Stay anchored to v0.2 commit

**Positive.** v0.3 opens immediately; the working-tempo pattern
that delivered v0.2 carries forward. The forward-compat contract
inherits the v0.2 triple invariant directly. Reviewability is
maximal — same `3b7b18af` text, same audit-gap-pin convention.
Items that don't fit the v0.2 envelope are explicitly out of
scope (clean boundary).

**Negative.** Candidates that require physics-scope expansion (e.g.
aggregation, full κ(T) refinement, wall hydrodynamics) are forced
out of v0.3 to a sibling track or a later cycle. The cycle is
narrower than the candidate menu suggests.

**Neutral.** This option re-uses ADR 0001's three upstream
recommendations *unchanged*; v0.3 is "v0.2 with implementation
tightenings under the same envelope."

### Option 3 — Hybrid

**Positive.** Maximises in-scope coverage of the candidate menu
without forcing a single decision on the spec authority question.

**Negative.** Bookkeeping cost: every v0.3 phase now records which
spec it anchors against. The forward-compat contract has to be
written per item, not once. Reviewers must validate phase-by-phase
that the right pin is consulted. Pattern 14 (forward-compatible
parameter splits) does not fully cover this — Pattern 14 keeps
*defaults* compatible; it does not silence the question of which
*spec* a non-default change shipped against. This is the option
ADR 0001 already considered (Option C there) and rejected for the
v0.2 cycle.

**Neutral.** Could be the right answer for a much larger scope than
v0.2 / v0.3 — the bookkeeping cost amortises across many phases.
For a 7-day cycle the overhead dominates.

## Working notes during deliberation

Recorded for context, not yet decided:

- **Stewardship preference (U. Warring, 2026-04-30):** lean toward
  **Option 2 with a hard rule** — repeat the v0.2 working pattern
  (~10 sessions over ~2 weeks against a frozen pin) rather than
  introducing calendar risk (Option 1) or per-phase bookkeeping
  overhead (Option 3). Reasoning: tempo, not authority, is the
  binding constraint at v0.3 scale.
- **Pattern 11** ("Spec pinning at commit-hash precision") plus
  **ADR 0001 precedent** both favour Option 2 if the candidate
  scope can be reshaped to fit the v0.2 envelope.
- **Open question for Option 2:** which v0.3 candidates actually
  *require* a physics-scope expansion vs. which can land as
  zero-default extensions like the v0.2 additions did? D3
  (aggregation) is the largest such item.

These working notes do not constitute the decision. They exist so
the deliberation has a stable record.

## Open questions

These remain open even within Option 2:

- **Q1.** If Option 2 is chosen, which v0.3 candidates need a parallel
  breakout track because they exceed the breakout-note v0.2 envelope?
  D3 (aggregation) is the canonical example; items E (wall
  hydrodynamics) and G (refined convection) may also qualify.
- **Q2.** What re-pin policy applies if breakout-note v0.3 lands
  mid-cycle? ADR 0001's "Phase 15 picks up the new pin" clause sets
  a precedent — does the v0.3 cycle adopt the same?
- **Q3.** If Option 1 is chosen, what is the timeout? At what point
  does indefinite upstream blocking convert into Option 2 by default?
- **Q4.** If Option 3 is chosen, what is the per-phase pin-recording
  format? Per-phase ADRs would be the natural tool, at the cost of
  multiplying the ADR count.

## Related but not decided here

- **D3 — aggregation pre-screen.** Whether DLVO `τ_agg(ζ, I, pH)`
  ships in-scope, as a parallel breakout, or deferred entirely is
  related to the spec-anchoring choice (it's most likely a parallel
  breakout under Option 2, in-scope under Option 1, ambiguous under
  Option 3) — but D3 has its own physics and effort considerations.
  D3 is decided in its own deliberation surface, not here.
- **D4 — wall hydrodynamics.** Same logic as D3.
- **D5 — convection refinement (κ(T) etc.).** Same logic.
- **D6 — `σ_geom` scan-axis promotion.** ADR 0001 explicitly
  delegated this to the breakout-note authors. Whatever they decide
  feeds back into Option 1 vs Option 2 here, but the mechanism is
  upstream and external to this ADR.

The boundary is deliberate: this ADR answers "what spec does v0.3
anchor against?" only. It does not answer "what items go in v0.3?"
