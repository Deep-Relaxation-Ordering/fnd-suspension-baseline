This ADR resolves **two coupled decisions** that govern the `pilot-v0.3`
cycle:

1. **D1 (spec anchoring) — Option 2.** `pilot-v0.3` stays anchored to
   breakout-note v0.2 commit `3b7b18af`, the same pin
   [ADR 0001](0001-v0.2-spec-anchoring.md) recorded for v0.2.
2. **First-slice selection — S2.** The first v0.3 implementation
   slice is **S2 — Stokes-Einstein corrections at sub-150-nm radii**,
   per [`docs/program-context.md` §3.1](../program-context.md). S1
   (DLVO aggregation pre-screen) follows once its upstream breakout
   note lands.

The two decisions are coupled rather than independent: Option 2 says
"v0.3 = implementation tightenings under the v0.2 spec envelope"; S2
is the *leading* such tightening because it has no upstream
dependency and the smallest blast radius of the candidate L1 slices.
Recording both in one ADR mirrors how the upstream rationale in
program-context.md presents them.

D2 (forward-compat baseline) and the remaining work-plan-v0-3
decisions (D3 / D4 / D5 / D6 / D7) are not resolved here. They get
their own deliberation surfaces inside the v0.3 cycle as they ripen.

## Context

[`pilot-v0.2`](../release-notes/v0.2.md) is closed. Its scope was
anchored under [ADR 0001](0001-v0.2-spec-anchoring.md) to
breakout-note v0.2 commit `3b7b18af`, with three forward-compatible
extensions landing as zero-default audit-gap pins. The
[`pilot-v0.2.1`](../release-notes/v0.2.md#fair-metadata-patch-021)
FAIR-metadata patch added citation infrastructure but did not
re-anchor the spec.

Phase 16.3 added [`docs/program-context.md`](../program-context.md)
v0.1 — the long-horizon goal document. It frames `pilot-v0.3` as a
*tactical sub-slice inside L1*, not the L1 release itself, and
identifies S1–S7 as the L1 slice menu. Critically, program-context
§3.1 / D-PC-2 records that **S2 ships first** as the
implementation-ready slice (no upstream dependency, smaller blast
radius), with S1 following once the upstream breakout lands.

Two questions then have to be answered before any v0.3 phase work
starts:

- **Which spec does v0.3 anchor against?** (work-plan-v0-3 §0 D1 —
  three options.)
- **Which slice ships first inside that anchor?** (program-context
  §3.1 reasoning; the work-plan-v0-3 §1 candidate menu A–J does not
  yet enumerate S2 as a discrete item.)

If either is left implicit, every per-phase decision in the v0.3
cycle inherits an unstated assumption — the pattern Pattern 11
("Spec pinning at commit-hash precision",
[findings-process.md §11](../findings-process.md#11-spec-pinning-at-commit-hash-precision))
exists to prevent.

## Decision 1 — Spec anchor: Option 2

`pilot-v0.3` stays anchored to breakout-note v0.2 commit `3b7b18af`.
Every v0.3 candidate must fit inside the existing breakout-note
envelope. Items that would require a physics-scope expansion are
pushed out of v0.3 to a parallel breakout track or a later cycle.

### Options considered

(Mirroring [`work-plan-v0-3.md` §0](../work-plan-v0-3.md#0-spec-anchoring-open) labels.)

- **Option 1 — Re-anchor to a future breakout-note v0.3.** `pilot-v0.3`
  opens only after the breakout-note repository
  (`Deep-Relaxation-Ordering/diamonds_in_water`) publishes a v0.3
  spec.
- **Option 2 — Stay anchored to breakout-note v0.2 commit `3b7b18af`.**
  v0.3 reuses the v0.2 spec pin verbatim.
- **Option 3 — Hybrid.** In-scope items that fit the v0.2 spec stay
  anchored to `3b7b18af`; items that require physics-scope expansion
  are split into parallel breakouts.

### Decision drivers

1. **Traceability.** Can a future reader unambiguously identify which
   spec a v0.3 artefact ships against? ADR 0001 set the precedent
   that this is non-negotiable.
2. **Reviewability.** Can the v0.3 cycle's per-phase reviewers see
   the spec text that justifies a phase's scope without leaving the
   repo?
3. **Forward-compat impact.** What does each option imply for the
   `pilot-v0.2.1` → `pilot-v0.3` cache and deliverable diffs?
4. **Risk of numeric-channel churn.** If the spec moves underneath
   the cycle (Option 1) or is split (Option 3), how likely is a §5
   cache regen mid-cycle?

| Driver | Option 1 | Option 2 | Option 3 |
|---|---|---|---|
| Traceability | Excellent once upstream lands; ambiguous until it does | Excellent and immediate | Acceptable; per-item pin records add overhead |
| Reviewability | Reviewer must context-switch to a sibling repo | Same `3b7b18af` text v0.2 reviewers read | Reviewer must check which item is anchored where |
| Forward-compat impact | Unknown | Same triple invariant as v0.2 | Mixed |
| Numeric-channel churn risk | Medium-to-high if upstream lands mid-cycle | Low (Phase 12.1 pattern reuses cleanly) | Medium |

### Why Option 2

- **Working tempo.** v0.2 took ~7 working days across ~10 sessions
  because it ran sequentially against a frozen spec. Gating v0.3 on
  an external publication (Option 1) breaks that tempo.
- **Reviewability.** Option 2 reuses the same `3b7b18af` text the v0.2
  reviewers read; Option 1 forces context-switching to a sibling
  repo at a moving pin; Option 3 multiplies per-item bookkeeping.
- **Numeric-channel safety.** The Phase 12.1 regression-audit pattern
  (`δ_shell = 0` reproduces v0.1 to machine precision) reuses cleanly
  under Option 2; Options 1 and 3 carry meaningful churn risk.
- **Program-context alignment.** [`docs/program-context.md` §3](../program-context.md)
  treats v0.3 as housekeeping inside L1, not as a layer-defining
  release. Tightenings under the existing envelope are exactly what
  Option 2 enables; layer-defining slices (S1 aggregation, S6 wall
  hydrodynamics) that exceed the envelope are deferred to their own
  upstream breakouts under the [§5 supplier-failure clause](../program-context.md#5-coupling-to-sibling-repositories).

### Consequences of Option 2

**Positive.** v0.3 opens immediately; the working-tempo pattern that
delivered v0.2 carries forward. The forward-compat contract inherits
the v0.2 triple invariant directly. Reviewability is maximal — same
`3b7b18af` text, same audit-gap-pin convention. Items that don't fit
the v0.2 envelope are explicitly out of scope (clean boundary).

**Negative.** Candidates that require physics-scope expansion (e.g.
S1 DLVO aggregation, full κ(T) refinement, S6 wall hydrodynamics)
are forced out of v0.3 to a sibling track or a later cycle. The
cycle is narrower than the work-plan-v0-3 candidate menu suggests.

**Neutral.** Option 2 re-uses ADR 0001's three upstream
recommendations *unchanged*; v0.3 is "v0.2 with implementation
tightenings under the same envelope."

### Re-pin policy if breakout-note v0.3 lands mid-cycle

ADR 0001's "Phase 15 picks up the new pin" clause sets the
precedent. If breakout-note v0.3 lands during the v0.3 cycle, the
cycle's release phase (proposed Phase 24) re-pins to the new commit
and records the pin movement in a Phase-25-style follow-up note.
Mid-cycle re-pinning is *not* allowed — it would invalidate the
in-flight slice's reviewability anchor.

## Decision 2 — First slice: S2

The first v0.3 implementation slice is **S2 — Stokes-Einstein
corrections at sub-150-nm radii** as defined in
[`docs/program-context.md` §3.1](../program-context.md).

### Why S2 first (and not S1)

S1 (DLVO aggregation pre-screen) and S2 (sub-150-nm SE corrections)
are both "must land before v1.0." The ordering choice is recorded
as D-PC-2 in program-context §7 and resolved as follows:

- **S1 needs an upstream breakout note** that does not yet exist
  (`Deep-Relaxation-Ordering/diamonds_in_water` owes it). Starting
  S1 first would gate the v0.3 cycle on a calendar item this repo
  cannot drive.
- **S2 has no upstream dependency.** It was adopted into Tier-1 from
  review R4 (Phase-9.x lab notes). Calibration data exists (Laloyaux
  z₂ tabulations of the gold-nanoparticle benchmark).
- **S2's blast radius is smaller.** It refines diffusivity given a
  trusted radius; it does not change the radius schema, the cache
  shape, or the regime-classifier inputs in ways S1 would.
- **Forward-compat is straightforward.** λ = 1.0 reproduces v0.2 to
  machine precision (the Pattern 14 zero-default extension shape
  v0.2 already uses for `δ_shell`, `delta_T_assumed`, and `σ_geom`).

The order is a **sequencing convenience, not an independence claim.**
S2 outputs are not absolutely trustworthy until S1 is in place — S2
refines the diffusivity *given* a trusted radius; S1 establishes
when the radius can be trusted. Section "API surface" below
encodes this constraint at the interface level so it cannot be
silently ignored.

### What S2 ships

Per program-context §3.1:

- λ-sweep over `{0.1, 0.5, 1.0}` of the SE breakdown coefficient.
- Calibration against the gold-nanoparticle benchmark (Laloyaux z₂
  tabulations).
- Forward-compatible default: `λ = 1.0` reproduces v0.2 to machine
  precision (same triple-invariant contract v0.2 used).

The S2 implementation phase (proposed Phase 18) is responsible for
the slice itself. This ADR records the *selection*, not the
implementation plan; the work plan does the rest.

### API surface — the `provisional=True` contract

To prevent well-meaning downstream users from treating S2-only
outputs as if L1 were complete:

- **Result objects from S2-refined diffusivity calculations carry a
  `provisional=True` flag** while S1 has not landed. The flag is
  metadata on the result object, not a soft warning, so it survives
  serialisation.
- **The design-tool entry points** (the planned `plan_*` helpers, or
  any future `pilot_v1.plan_experiment`-style API) **refuse to
  consume `provisional=True` outputs** without an explicit
  `accept_provisional=True` override.
- **The override is loud, not silent.** It must be passed at the
  call site — no default, no environment variable, no config file.
  Users who pass it accept the responsibility that S1's
  trustworthiness flag is not yet enforced on their inputs.
- **The flag flips off automatically when S1 lands.** S1's slice is
  responsible for clearing `provisional=True` (or, equivalently,
  introducing a separate trustworthiness channel that the design-
  tool entry points consult instead of the provisional flag).
  Whichever shape S1 chooses, the v0.3 → v1.0 transition is
  responsible for removing the override pathway.

The exact module / function / dataclass surface that carries the
flag is a Phase 18 implementation question, deliberately not pinned
here. What this ADR pins is the *contract*: provisional outputs
exist, the design-tool refuses them by default, and the override is
explicit.

## Working notes that fed the deliberation

Recorded for context, retained from the `Proposed (stub)` draft
because the rationale is now load-bearing under Decision 1:

- **Stewardship preference (U. Warring, 2026-04-30):** Option 2 with
  a hard rule — repeat the v0.2 working pattern (~10 sessions over
  ~2 weeks against a frozen pin) rather than introducing calendar
  risk (Option 1) or per-phase bookkeeping overhead (Option 3).
- **Pattern 11** ("Spec pinning at commit-hash precision") plus
  **ADR 0001 precedent** both favour Option 2.
- **Three independent reviews** of the [program-context document](../program-context.md)
  draft converged on "S2 first, S1 follows" — recorded as resolved
  D-PC-2.

## Open questions (acknowledged, not blocking)

These remain open inside Option 2 and are routed to other
deliberation surfaces:

- **Q1.** Which v0.3 candidates need a parallel breakout track
  because they exceed the breakout-note v0.2 envelope? S1
  (aggregation) is the canonical example; S6 (wall hydrodynamics)
  may also qualify. Decided per item in work-plan-v0-3 §1 reshape
  during the Phase 17 ceremony's continuation.
- **Q2.** Once S1 lands and `provisional=True` is no longer needed,
  does the override pathway get removed entirely, or kept as a
  testing affordance? Decided in the v0.3 → v1.0 transition ADR
  (currently TBD).
- **Q3.** Forward-compat baseline (work-plan-v0-3 §3 D2): anchor to
  `pilot-v0.2.1` or to a v0.3 baseline? S2's `λ = 1.0` default
  reproduces v0.2.1 to machine precision, which favours D2 = Option 1
  for S2 specifically. Other slices (S6, refined convection) may
  force D2 = Option 2. Decided per item, not here.

## Related but not decided here

- **D3 — aggregation in/out (work-plan-v0-3 §1 item D / program-context S1).**
  Under Option 2, S1's full implementation requires an upstream
  breakout note and is therefore *not* in v0.3's scope. v0.3 may
  prepare scaffolding (parameter slots, API stubs) but not the
  trustworthiness flag itself.
- **D4 — wall hydrodynamics (program-context S6).** Same logic;
  out of v0.3 unless its breakout-note specifications fit the v0.2
  envelope, which currently they do not.
- **D5 — convection refinement (κ(T) etc.).** Whether T-dependent
  thermal diffusivity fits the v0.2 envelope is a per-knob question
  and is decided in its own deliberation surface.
- **D6 — `σ_geom` scan-axis promotion.** ADR 0001 explicitly
  delegated this to the breakout-note authors. Whatever they decide
  feeds back into Option 1 vs Option 2 here, but the mechanism is
  upstream and external to this ADR.
- **D7 — Development Status classifier.** Carries forward
  unchanged; v0.3 stays at `3 - Alpha` per the precedent that
  promotion to `4 - Beta` happens at v1.0 (program-context §4.1).

The boundary is deliberate: this ADR answers "what spec does v0.3
anchor against, and what slice ships first inside that anchor?"
only.
